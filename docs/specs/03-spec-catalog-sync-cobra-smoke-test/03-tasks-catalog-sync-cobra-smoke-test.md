### Notes

- Tests live next to the file they cover (`cmd/catalog_sync_test.go` next to `cmd/catalog_sync.go`) — matches the `cmd/catalog_add_test.go` precedent.
- Use the standard `testing` package; AAA layout; table-driven only where it reduces duplication without obscuring intent.
- Run tests with `go test ./...` (or `go test -run TestRunCatalogSync ./cmd/...` for the suite-under-development).
- **Strict TDD per `CLAUDE.md`:** every new test must be authored in failing state and then made pass — no writing production code ahead of the test that needs it.
- The first test's RED state is **build failure**, not test failure: it references a `runCatalogSyncWithDeps` symbol that doesn't exist yet. That's a valid RED in Go.
- Concurrency assertion in 1.8 needs a fake `Fetcher` with `atomic.Int32` counters (`inflight`, `peak`) and a synchronization knob so multiple fetches stay simultaneously in-flight long enough to observe the cap.
- The golden test must be **deterministic** despite parallel workers — sort the captured lines before comparing, or force concurrency=1 inside the test, or sequence the fake fetcher's completions. Sort-before-compare is simplest.
- Conventional-commit prefix: `test(catalog):` (with the refactor folded in) or split as `refactor(catalog):` + `test(catalog):` if the refactor is large enough to deserve its own review. Default to one commit.
